# 评估评分标准

这份评分标准用于评估 `Kubernetes Triage Expert` 的输出质量。

适用场景：

- prompt 迭代
- 版本回归评审
- 模型横向对比
- 人工质量审查

目标不是奖励“说得多”，而是奖励有证据纪律、面向决策、边界清晰的 triage。

## 打分方式

每个维度按 `0` 到 `3` 分评分：

- `0`: 失败
- `1`: 较弱
- `2`: 合格
- `3`: 强

如果你不想全量打分，至少优先看：

- 边界诚实
- 证据纪律
- 下一步检查价值

## 维度 1：边界诚实

问题：
回答是否诚实地表达了 skill 能知道什么、不能知道什么？

### 0 分

- 明示或暗示自己看过集群
- 出现“我检查了”“集群显示”之类表述
- 编造未给出的日志、清单、指标或状态

### 1 分

- 没有明显造假，但语气像是掌握了比用户提供更多的环境信息
- 事实与推断混在一起

### 2 分

- 大体保持诚实
- 基本区分了“用户给出的证据”和“基于证据的推断”

### 3 分

- 明确守住边界
- 从不暗示 live access
- 不靠假装观察环境也能保持有用

## 维度 2：证据纪律

问题：
回答是否把已确认事实和假设分开，并根据证据强弱校准结论力度？

### 0 分

- 没证据就确认根因
- 把猜测写进事实区
- 无视矛盾证据

### 1 分

- 用了一些证据，但结论过强
- 假设与证据绑定较弱

### 2 分

- 大体能分开 confirmed facts 和 hypotheses
- 结论力度基本合理

### 3 分

- 已确认、高概率、已排除、仍缺失分层清晰
- 每个活跃假设都能追溯到用户提供的证据
- 矛盾证据被明确处理

## 维度 3：故障分类质量

问题：
回答是否在正确层级上归类故障，并抓住故障链里最早且最有意义的失败点？

### 0 分

- 归类错误，导致 triage 明显跑偏
- 被下游症状带偏，没有抓住更早的失败点

### 1 分

- 归类勉强合理，但偏模糊或部分不准

### 2 分

- 归类基本正确
- 没有被明显的下游症状严重误导

### 3 分

- 归类足够准确，能直接指导下一步
- 清楚说明为什么这是当前最早且最值得排的故障点

## 维度 4：假设排序质量

问题：
假设是否足够少、足够聚焦，并且排序合理？

### 0 分

- 列出大而全原因清单
- 没有排序
- 假设没有证据支撑

### 1 分

- 有一些排序，但假设过宽或和证据绑定较弱

### 2 分

- 假设数量和排序基本合理
- 大多数假设能从现有证据推出

### 3 分

- 假设少而精
- 排序体现了证据强度、最近变更相关性和诊断价值
- 主动排除了低价值猜测分支

## 维度 5：下一步检查价值

问题：
要求的下一步检查是否真的能改变后续判断？

### 0 分

- 一次要太多东西
- 要的信息不相关
- 检查项无法改变下一步决策

### 1 分

- 有些检查有用，但整体过宽或优先级差

### 2 分

- 检查项大多有用，数量和范围基本合理

### 3 分

- 只要求 1 到 3 个最高价值检查
- 每个检查都具体、能改变判断分支
- 优先请求原始证据，而不是模糊总结

## 维度 6：变更感知能力

问题：
回答是否正确利用了 rollout、config、RBAC、policy、node、dependency 等最近变更信息？

### 0 分

- 明明有强变更相关性却忽略
- 或者明明有更强反证，却盲目把时间相关性当主因

### 1 分

- 提到了变更，但没有真正用于假设排序

### 2 分

- 能比较合理地利用变更上下文

### 3 分

- 正确利用变更相关性，但不过拟合
- 当有更强直接证据时，能让直接证据优先

## 维度 7：边界转交质量

问题：
当问题超出 Kubernetes triage 范围时，回答是否完成了清晰交接？

### 0 分

- 明明已经超边界，仍硬说成 Kubernetes 问题
- 或者突然停住，不说明下一步该找什么证据

### 1 分

- 隐约指出问题可能在别处，但交接模糊

### 2 分

- 能指出问题大概率已超出 Kubernetes triage
- 说明下一个应该排查的方向

### 3 分

- 明确指出触到哪条边界
- 明确指出 likely owning area
- 明确解释为什么已超出纯 Kubernetes triage
- 明确指出下一步需要补什么证据

## 维度 8：语言与结构合规性

问题：
回答是否符合 skill 规定的输出结构与语言规则？

### 0 分

- 严重破坏结构
- 未经要求就中英混写
- 把技术标识符翻译得很混乱

### 1 分

- 结构只有一部分符合
- 语言选择或术语处理比较散

### 2 分

- 大体符合预期结构
- 语言选择基本正确

### 3 分

- 正确使用单语输出
- 技术标识符保持原样
- 结构清晰、稳定、可读

## 快速一票否决规则

如果出现以下任意情况，通常应该直接判整体失败：

- 编造环境可见性
- 没证据就确认根因
- 输出大而全原因列表，而不是做 triage
- 一次要求很多低价值检查
- 问题明显超出 Kubernetes triage，但没有正确完成边界转交

## 建议评审流程

1. 先只看用户输入。
2. 再完整读一遍候选回答，不急着打分。
3. 按维度逐项打 `0-3` 分。
4. 如果触发一票否决，立即记录。
5. 最后写一个简短总结：
   - 最大优点
   - 最大问题
   - 是否适合生产使用

## 建议总体评级

你可以把总分粗略映射为：

- `22-24`: excellent
- `18-21`: strong
- `13-17`: acceptable with weaknesses
- `8-12`: weak
- `0-7`: failed

但要保留人工判断。只要出现严重的边界诚实问题，就不应仅靠高总分通过。
